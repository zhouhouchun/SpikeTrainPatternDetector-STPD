
# Architecture Notes

## Why this refactor exists

The earlier single-file prototype reached a maintenance boundary. It contained detector logic, UI, exports, QC, ML, and governance in one file. This package separates those concerns while preserving reference detector behavior.

## Current modularization level

This package is a transitional modularization. It splits source files, adds a native C backend, routes compatibility behavior through semantic implementation names, materializes defaults from `inst/config/parameters.yml`, and generates the main detector-parameter UI from the YAML parameter contract.

## Native C backend

The C backend currently accelerates:

1. Per-train ISI percentile calculation.
2. Per-train local-median cache calculation.

These are high-frequency operations that affect visualization, seed detection, local-compression candidates, and pause context estimation.

## Future work

- Continue moving specialized UI sections onto schema-driven helpers where they are not genuinely interactive workflows.
- Move candidate feature extraction into a single table used by classification, ledger, export, hover, and comparison.
- Add formal test fixtures for boundary bursts, refractory doublets, high-frequency tonic/spiking, long_burst, pause near HF packets, candidate/event ledger split, and CSV export.
- Move structure scan and interval joins to C/Rcpp only after the R module boundaries are stable.

## schema-driven refactor notes

The canonical public API now avoids version suffixes:

- `default_params()`
- `run_detector_dataset()`
- `run_detector_train()`
- `compute_candidate_feature_table()`
- `final_classify_candidate()` / `final_classify_candidates()`
- `scan_structure_candidates()`
- `interval_best_overlap()`
- `export_detection_results()`

Historical suffix-bearing names have been removed from current code paths. Future development should keep detector internals on the canonical candidate-generation -> feature-table -> final-classification -> ledger -> export pipeline.


## Public API consolidation layer

The public API routes detector runs through `stpd_detect()`, adds result consistency checks, and exports strict/candidate-family event-level validation reports when MANUAL labels are available. Detector semantics are unchanged.

## Public audit, CSV input, and parameter namespace policy

The event-grammar detector keeps diagnostic candidate windows and public biological calls separate.
Rejected, profile, unwritten, blocked or not-selected windows stay in `Candidate_diagnostic_audit.csv` and are not propagated into `Candidate_ledger.csv`, `Eventness_audit.csv` or final classification exports.

`rejection_reason` is reserved for diagnostic/rejected rows. Public selected candidates should leave it blank; neutral external strings such as `no rejection`, `not rejected` and `not applicable` are treated as neutral.

Raw CSV import expects spike timestamp columns. High-confidence derived tables such as `Sliding_*`, `ISI_base`, `tonic_summary`, `threshold`/`threshould`, `Candidate_ledger`, `Eventness_audit`, `Events_final`, `diagnostic candidate`, `summary`, `result`, `features` and `audit` files are blocked by default. Use `allow_derived_csv=TRUE` only for intentional overrides.

## Duplicate timestamp policy

Exact duplicate timestamps are QC-relevant because they create zero or near-zero ISIs that can affect burst/pause, LV/CV/MM and high-frequency statistics. Formal analyses should either use `duplicate_policy="collapse_exact"` or report duplicate prevalence and sensitivity analyses.


## Product parameter namespace

Public configuration should use `params$spiketrainpattern`. Compatibility handling for older saved workspaces is resolved by `stpd_productize_params()` before detection and should not be used as a public parameter namespace.

Default runtime parameters, product defaults, key UI schema rows, eventness-audit schema rows, and the full `parameter_contract` now live in `inst/config/parameters.yml`. `default_params_sec()`, `stpd_product_schema_defaults()`, `stpd_key_parameter_schema()`, `stpd_parameter_contract()`, and the eventness schema extension all read from that YAML source. `stpd_parameter_registry()` and `stpd_validate_params()` use the contract for type, range, choice, and UI metadata validation/reporting, so parameter edits should be made there first and then verified with the schema synchronization tests.

The Shiny `Detector / Parameters` tab is generated by `stpd_contract_ui_controls()` from `parameter_contract`. Dedicated workflow controls remain hand-built for data import/QC, manual annotation, visualization, train-specific calibration, and detector-family toggles; generated contract controls use the `contract_param_` input namespace and are merged into `read_params_from_ui()` after the quick schema controls. The contract now carries `ui_level`, ordering, section, visibility/help, and control metadata; the UI exposes Basic / Advanced / Expert / All filters with Basic as the default. Basic labels and help text are written for biological interpretation and ordered through the normal analysis workflow. The same tab provides parameter YAML import/export, current-UI contract validation, a UI-visible parameter change impact preview, a selected-train dry-run event-difference preview via `stpd_parameter_delta_preview()`, click-through changed-event navigation into the aligned raster, raster/ISI-profile overlays produced from `stpd_parameter_delta_overlay_rows()`, CSV bundle export via `stpd_parameter_delta_export()`, and a YAML write/read hash round-trip check via `stpd_write_params_yaml()`, `stpd_read_params_yaml()`, and `stpd_parameter_yaml_roundtrip_report()`. The Scientific validation tab uses `stpd_event_level_validation_report()` and `stpd_parameter_sensitivity_scan()` to connect Basic-parameter perturbations to event-level MANUAL agreement, boundary errors, label confusions, and method-record CSV exports.


## Version-neutral internal API

New detector internals should call the version-neutral API in `R/50_internal_api.R` rather than versioned implementation functions. The stable internal entry points include `stpd_detect_dataset_core()`, `stpd_detect_train_core()`, `stpd_detect_train_event_grammar()`, `stpd_event_grammar_params()`, `stpd_resolve_thresholds_for_dataset()`, and `stpd_candidate_audit_to_ledger()`.

The canonical result fields are `results$threshold_table`, `results$candidate_diagnostic_audit`, `results$candidate_features`, `results$final_decisions`, `results$eventness_audit`, and `results$run_metadata_public`. Normal detector results now drop historical versioned result fields after stable aliases are created. Compatibility logic remains private and should not be used by new code.


## Productized detection safeguards

The product schema layer applies the final detector hardening pass after the lower-level detector modules have loaded. This keeps private compatibility internals available while ensuring public parameters, QC policy, dataset-level thresholds, and manual-lock semantics are enforced at the stable API boundary.

Pre-detection QC runs before any pattern labels are generated. By default, data-integrity errors such as exact duplicate timestamps, zero-or-negative ISIs, and hard artifact ISIs stop detection before burst, pause, tonic, or high-frequency evidence is computed. For diagnostic continuation, set `params$spiketrainpattern$engine$stop_on_qc_error = FALSE`; formal analyses should collapse exact duplicates at import or report duplicate prevalence and sensitivity analyses.

Dataset/manual/histogram thresholds are resolved once at the dataset entry point and then frozen for all selected trains in that run. The resolved threshold table is stored in `results$threshold_table`, and the pre-detection QC table is stored in `results$pre_detection_quality`, so downstream exports can audit the data-integrity gate and the effective threshold policy.

The public `lock_manual` meaning is strict: with `lock_manual = TRUE`, manual labels remain final-label dominant and AUTO labels are not written onto manually labeled ISIs. AUTO evidence can still be generated on unlocked intervals for diagnostic review.
